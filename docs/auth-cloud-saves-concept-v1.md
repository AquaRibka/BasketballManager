# Auth And Cloud Saves Concept v1

## Цель документа

Этот документ фиксирует черновую концепцию пользователей, авторизации и облачных сохранений для этапа `Post-MVP`.

Его задача:

- описать, как добавить `User model` после стабильного локального MVP;
- зафиксировать базовый `auth flow` без переписывания текущей модели `CareerSave`;
- определить правила `save ownership`;
- обозначить минимальные `security basics`;
- не втягивать аккаунты и облачную синхронизацию в текущий MVP раньше времени.

## Продуктовая роль auth и cloud saves

После стабилизации локального MVP аккаунты нужны не ради самой авторизации, а ради устойчивого владения прогрессом.

Их смысл:

- позволить пользователю вернуться к своей карьере на другом устройстве;
- связать несколько сохранений с одним владельцем;
- подготовить базу для будущих онлайн-функций;
- не терять простоту локального старта игры.

Система должна отвечать на простой вопрос:

```text
Как сохранить удобство локальной одиночной игры
и при этом дать пользователю надежное облачное владение карьерой?
```

---

## Базовый принцип первой версии

Первая версия auth/cloud saves должна быть надстройкой над существующей карьерной моделью, а не ее заменой.

Ключевое правило:

```text
CareerSave остается курсором карьеры.
Auth добавляет владельца и доступ,
но не превращает save в новый доменный монолит.
```

Это означает:

- текущая логика `CareerSave` сохраняется;
- сезонные данные по-прежнему живут в своих доменных сущностях;
- ownership добавляется как отдельный слой поверх текущего save lifecycle.

---

## Почему нельзя ломать текущий CareerSave

В текущей концепции `CareerSave` хранит:

- имя сохранения;
- выбранную команду;
- активный сезон;
- текущий курсор прогресса.

Это удачная граница ответственности.

Если при добавлении auth начать складывать в `CareerSave`:

- профиль пользователя;
- auth-state;
- sync metadata;
- device mapping;
- историю логинов,

то save-модель потеряет простоту и станет смешивать разные домены.

Поэтому post-MVP подход должен быть таким:

- `CareerSave` остается сущностью игрового прогресса;
- `User` становится сущностью идентичности;
- связь между ними описывает владение и доступ.

## User Model

### Зачем нужен отдельный пользователь

Отдельный `User` нужен, чтобы:

- иметь устойчивый идентификатор владельца;
- привязывать к нему несколько карьер;
- поддержать future login/session management;
- не смешивать игрока как человека и команду как игровой выбор.

### Минимальный состав `User` для v1

Для первой версии достаточно простого набора:

- `id`
- `email`
- `passwordHash` или внешний auth identifier
- `displayName`
- `createdAt`
- `updatedAt`
- `lastLoginAt`

### Важный принцип

`User` не должен знать подробности симуляции.

Он отвечает только за:

- identity;
- authentication;
- ownership;
- базовые account preferences в будущем.

---

## Auth Flow

### Базовая продуктовая стратегия

Рекомендуемый порядок развития:

1. локальный anonymous MVP;
2. post-MVP optional account creation;
3. затем cloud-linked saves;
4. только потом возможный cross-device default flow.

Это позволяет не блокировать релиз игры из-за auth-инфраструктуры.

### Рекомендуемый flow первой версии

Для `Post-MVP v1` достаточно простого сценария:

- `sign up`
- `sign in`
- `sign out`
- `restore session`
- `attach local save to account` или создать новый cloud save под аккаунтом

### Практический смысл

Пользователь должен иметь возможность:

- играть локально без аккаунта;
- позже создать аккаунт;
- привязать выбранное сохранение к пользователю;
- после этого видеть свои saves на другом устройстве.

### Что важно избежать

Не стоит сразу делать:

- обязательную авторизацию на старте;
- сложную multi-device conflict resolution;
- социальные логины как обязательную часть системы;
- переписывание current save endpoints вокруг auth еще до появления ownership model.

## Save Ownership

### Главный вопрос

После появления аккаунтов система должна явно отвечать:

- кому принадлежит это сохранение;
- кто может его читать;
- кто может его изменять.

### Рекомендуемая первая модель

Для первой версии достаточно single-owner модели:

- у каждого `CareerSave` может быть `ownerUserId`;
- save либо локальное/анонимное, либо привязано к одному пользователю;
- совместный доступ и shared franchises не нужны.

### Как сохранить совместимость с текущим MVP

Есть два безопасных варианта:

#### Вариант A: nullable ownership в `CareerSave`

Добавить в `CareerSave` поле:

- `ownerUserId?: string | null`

Плюсы:

- простая интеграция;
- минимум join complexity;
- текущая save-сущность почти не меняется.

#### Вариант B: отдельная таблица ownership

Создать связующую сущность:

- `CareerSaveOwnership`

С полями:

- `careerSaveId`
- `userId`
- `createdAt`

Плюсы:

- чище разделены домены;
- легче расширять доступ в будущем.

Минусы:

- больше инфраструктурной сложности для первой версии.

### Рекомендуемое решение

Для `Post-MVP v1` разумнее начать с `ownerUserId` в `CareerSave`.

Почему:

- это не ломает базовую идею save как курсора;
- ownership остается маленьким и понятным расширением;
- migration path к отдельной access-таблице сохраняется на будущее.

## Cloud Save Strategy

### Что считать облачным сохранением

Облачное сохранение — это не новый вид игровой логики, а тот же `CareerSave`,
доступный через backend под контролем ownership.

То есть различие такое:

- локальный save может жить без пользователя;
- cloud save принадлежит `User` и читается через авторизованный API.

### Базовая стратегия синхронизации

Для первой версии лучше выбрать простую модель:

- один backend считается source of truth для cloud saves;
- клиент не делает сложных merge-операций;
- при конфликте выигрывает явно выбранное действие пользователя.

### Практически это означает

На старте достаточно поддержать:

- `upload local save to account`;
- `load owned saves`;
- `continue owned save`;
- `delete owned save`.

Автоматическую двустороннюю sync-механику лучше отложить.

## Security Basics

### Минимальные требования

Даже первая версия auth должна закрывать базовые риски.

Минимум для `Post-MVP v1`:

- хранить не пароль, а `passwordHash`;
- использовать безопасную hash strategy вроде `argon2` или `bcrypt`;
- проверять ownership на каждом `GET/POST/DELETE` для конкретного save;
- не отдавать чужие сохранения в `GET /saves`;
- использовать короткоживущую session/token стратегию;
- иметь базовую rate limiting защиту на auth endpoints;
- не возвращать лишние детали о том, существует ли конкретный email;
- вести базовый audit trail для sign-in и save ownership changes.

### Что особенно важно для saves

Самая критичная ошибка здесь — не слабый UI, а неправильная авторизация доступа к save.

Поэтому правило должно быть жестким:

```text
Любое действие с cloud save
проверяет user identity и ownership
до обращения к карьерным данным.
```

## API Draft

### Новые доменные направления

После MVP могут появиться такие группы endpoint-ов:

- `POST /api/v1/auth/sign-up`
- `POST /api/v1/auth/sign-in`
- `POST /api/v1/auth/sign-out`
- `GET /api/v1/auth/session`
- `GET /api/v1/me`

### Эволюция saves API

Текущий `saves` API не нужно ломать полностью.

Его можно развивать так:

- локальный режим: текущие endpoints продолжают работать в dev/local MVP flow;
- cloud режим: те же save endpoints начинают фильтроваться по владельцу;
- для attach/import можно добавить отдельный endpoint.

Например:

- `POST /api/v1/saves/import-local`
- `POST /api/v1/saves/:id/claim`

Или один более явный вариант:

- `POST /api/v1/me/saves`

### Важный принцип контракта

API должно ясно различать:

- identity endpoints;
- ownership rules;
- игровые save endpoints.

Но при этом не дробить одну и ту же карьерную сущность на несколько несовместимых моделей.

## Data Model Draft

### Минимальный набор сущностей

Для первой auth/cloud-save версии достаточно следующих моделей:

- `User`
- `Session` или token-backed session store
- `CareerSave` с `ownerUserId?: string | null`

Опционально позже:

- `AuthAuditLog`
- `PasswordResetToken`
- `EmailVerificationToken`
- `DeviceSession`

### Пример минимального расширения

```text
User
- id
- email
- passwordHash
- displayName
- createdAt
- updatedAt
- lastLoginAt

CareerSave
- id
- saveName
- selectedTeamId
- currentSeasonId
- currentDate
- currentRound
- ownerUserId?  <-- new, nullable for backward compatibility
- createdAt
- updatedAt
```

## UI Draft

### Что нужно пользователю

UI post-MVP должен решать четыре простые задачи:

- создать аккаунт;
- войти;
- понять, где локальные и где облачные сохранения;
- привязать или загрузить нужную карьеру.

### Минимальные экраны и состояния

Для первой версии достаточно:

- `Sign In / Sign Up`
- `Profile / Session status`
- `Save list` с пометками `local` и `cloud`
- действие `Save to account`
- действие `Load from cloud`

### Важный UX-принцип

Пользователь не должен гадать:

- это сохранение только на устройстве;
- оно уже загружено в аккаунт;
- или попытка облачного сохранения завершилась ошибкой.

Статус владения и источник save должны быть явно видны.

---

## Ограничения первой версии

В `Post-MVP v1` не стоит включать:

- multiplayer ownership;
- shared teams или co-op careers;
- real-time sync между несколькими устройствами;
- merge двух разных веток одного save;
- сложную offline-first conflict resolution;
- social auth как обязательный путь входа.

Это полезные направления, но они не нужны для первого шага.

---

## Rollout

Рекомендуемая поэтапная раскладка:

1. стабилизировать локальный `CareerSave` flow;
2. ввести `User` и базовую session auth;
3. добавить nullable ownership для `CareerSave`;
4. ограничить список cloud saves по владельцу;
5. добавить UI для привязки локального save к аккаунту;
6. только после этого расширять sync-удобства.

Такой rollout снижает риск того, что auth перепутается с базовой игровой логикой.

---

## Критерий готовности

Концепт считается достаточным для backlog-проработки, если команда согласна с такими принципами:

- локальный MVP живет без обязательного аккаунта;
- `CareerSave` не переписывается в auth-centric сущность;
- ownership добавляется как тонкий слой над текущей моделью;
- cloud saves доступны только владельцу;
- security basics закрывают базовые риски уже в первой версии.

---

## Итог

`Auth/cloud saves` должны появиться только после стабильного локального MVP и быть эволюцией текущей системы, а не ее перезапуском.

Первая версия должна быть намеренно простой:

- отдельный `User`;
- базовый `auth flow`;
- понятный `save ownership`;
- минимально достаточная security-модель;
- сохранение текущей роли `CareerSave` как курсора карьеры.
