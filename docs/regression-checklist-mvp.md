# MVP Regression Checklist

## Цель документа

Этот документ нужен для ручной проверки MVP перед демонстрацией.
Он помогает быстро пройти критичный пользовательский сценарий и понять,
что приложение не выглядит сломанным на старте.

Проверяем цепочку:

- запуск приложения;
- просмотр команд;
- просмотр состава;
- симуляцию матча или раунда;
- обновление таблицы;
- восстановление состояния после перезагрузки.

---

## Подготовка

Перед проверкой должны быть доступны:

- локальная PostgreSQL;
- backend API;
- frontend web app;
- заполненная база demo или test seed.

Базовые команды запуска:

```bash
npm run dev --workspace @basketball-manager/backend
npm run dev --workspace @basketball-manager/frontend
```

Если нужно пересобрать данные:

```bash
npm run prisma:seed:test
```

Если нужен полный reset test-данных:

```bash
npm run prisma:reset:test
```

## Smoke Check

### 1. Run app

Шаги:

1. Запустить backend.
2. Запустить frontend.
3. Открыть приложение в браузере.

Ожидаемый результат:

- frontend открывается без белого экрана;
- основные разделы навигации видны;
- страницы не падают сразу в error state;
- backend отвечает без `500` на стартовые запросы.

Если проверка не пройдена:

- зафиксировать URL, экран и текст ошибки;
- проверить backend logs;
- проверить `.env` и подключение к БД.

---

### 2. View teams

Шаги:

1. Перейти в раздел `Команды`.
2. Убедиться, что список команд загружается.
3. Открыть любую команду из списка.

Ожидаемый результат:

- список команд виден и читается без пояснений;
- у команды открывается detail page;
- в header команды есть базовая информация;
- нет бесконечной загрузки и пустых карточек без текста.

Если проверка не пройдена:

- сохранить id команды, если он виден в URL;
- проверить ответ `GET /teams` и связанный API error.

---

### 3. View roster

Шаги:

1. На странице команды просмотреть таблицу состава.
2. Проверить сортировку или фильтр, если они доступны.
3. Выбрать игрока и открыть его блок деталей.

Ожидаемый результат:

- roster table содержит игроков;
- у игроков видны `name`, `position`, `overall` и атрибуты;
- player details обновляются при выборе другого игрока;
- layout не ломается на длинных именах или пустых состояниях.

Если проверка не пройдена:

- отметить, на каком игроке или фильтре проблема воспроизводится;
- приложить скрин и текст UI-сообщения.

---

### 4. Simulate match or round

Шаги:

1. Перейти в раздел `Календарь`.
2. Найти текущий раунд.
3. Нажать действие симуляции для текущего раунда.
4. Открыть один из сыгранных матчей.

Ожидаемый результат:

- календарь загружается по раундам;
- до симуляции матчи имеют статус `SCHEDULED`;
- после симуляции матч получает статус `COMPLETED`;
- у матча появляются счёт и `playedAt`;
- страница матча показывает home/away, score, status и basic stats, если они есть.

Если проверка не пройдена:

- сохранить id сезона и id матча из URL;
- проверить ответ симуляции и текст ошибки в UI;
- отдельно отметить, сломалась ли симуляция или только отображение результата.

---

### 5. Check standings

Шаги:

1. После симуляции перейти в раздел `Таблица`.
2. Сравнить таблицу с результатом сыгранного матча.

Ожидаемый результат:

- standings page открывается без ошибок;
- у команд отображаются `position`, `W-L`, `PF`, `PA`, `point diff`;
- хотя бы у двух команд изменяются `games played`, победы или поражения;
- данные выглядят консистентно со счётом симулированного матча.

Если проверка не пройдена:

- сохранить сезон и команды, по которым видна рассинхронизация;
- сверить match result и standings response.

---

### 6. Reload save

Шаги:

1. После симуляции обновить страницу браузера.
2. Снова открыть `Календарь`, `Таблицу` и страницу выбранной команды.
3. Если в UI есть явный сценарий создания карьеры или сохранения, повторно открыть текущую карьеру.

Ожидаемый результат:

- после reload приложение не возвращается в невалидное состояние;
- текущий сезон, календарь и standings остаются согласованными;
- сыгранный матч не откатывается обратно в `SCHEDULED`;
- выбранная карьера или save загружается без ручного восстановления данных.

Если проверка не пройдена:

- зафиксировать, что именно потерялось после reload:
  `season`, `schedule`, `standings` или выбранная команда;
- проверить endpoint загрузки save и связанные frontend-запросы.

---

## Минимальный критерий готовности

MVP можно показывать, если подряд проходят все проверки:

1. приложение запускается;
2. команды и состав открываются;
3. раунд симулируется без `500`;
4. таблица обновляется после симуляции;
5. после reload состояние не теряется.

---

## Итог

Этот checklist покрывает минимальный regression-проход перед демо MVP.
Если любой из пунктов падает, релизный сценарий считается нестабильным,
даже если отдельные API endpoints по отдельности работают.
