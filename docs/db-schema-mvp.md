# DB Schema MVP

## Цель документа

Этот документ кратко описывает текущую Prisma-схему проекта Basketball Manager для MVP-этапа.
Его задача — быстро объяснить:

- какие сущности уже есть в базе данных;
- как они связаны между собой;
- какие ограничения уже действуют;
- какими миграциями это создано.

---

## Текущий состав схемы

Сейчас в Prisma-схеме есть:

- `Team`
- `Player`
- `Season`
- `Match`
- `Standing`
- `MatchPlayerStat`
- `CareerSave`
- enum `PlayerPosition`
- enum `MatchStatus`
- enum `SeasonStatus`

Это минимальная база для MVP `Data/API`, достаточная для:

- хранения команд;
- хранения состава команды;
- отображения карточки команды;
- отображения игроков;
- хранения сезонов и матчей;
- хранения состояния карьеры пользователя;
- дальнейшего расчёта `overall` и симуляции.

## Entities

## Team

Модель: [prisma/schema.prisma](/home/newuser/Documents/BM/prisma/schema.prisma:11)

### Назначение

`Team` хранит клуб, которым можно управлять в карьере и который участвует в сезоне.

### Основные поля

- `id` — строковый идентификатор, Prisma `cuid()`
- `name` — полное название команды
- `city` — город команды
- `shortName` — короткое уникальное имя команды
- `rating` — общий рейтинг команды
- `createdAt` — дата создания записи
- `updatedAt` — дата последнего обновления

### Что даёт в MVP

- список команд для выбора и отображения;
- карточку команды;
- базовую оценку силы клуба;
- точку привязки для состава игроков.

## CareerSave

Модель: [prisma/schema.prisma](/home/newuser/Documents/BM/prisma/schema.prisma:131)

### Назначение

`CareerSave` хранит долгоживущее состояние сохранённой карьеры пользователя.

### Основные поля

- `id` — строковый идентификатор, Prisma `cuid()`
- `saveName` — название сохранения
- `selectedTeamId` — ссылка на выбранную команду пользователя
- `currentSeasonId` — ссылка на текущий сезон этой карьеры
- `currentDate` — текущая календарная дата внутри сохранения
- `currentRound` — текущий раунд/шаг сезона
- `createdAt` — дата создания сохранения
- `updatedAt` — дата последнего обновления сохранения

### Relations

- `CareerSave -> Team`
  `selectedTeamId` -> `Team.id`
- `CareerSave -> Season`
  `currentSeasonId` -> `Season.id`

### Поведение связей

- удаление выбранной команды запрещено, пока на неё ссылается сохранение
- удаление текущего сезона запрещено, пока на него ссылается сохранение
- изменение `Team.id` и `Season.id` каскадно обновляет связи

### Save state

Модель уже позволяет:

- создать сохранение с выбранной командой и стартовым сезоном;
- загрузить сохранение вместе с его связями;
- обновлять ход карьеры через `currentDate`, `currentRound`, `currentSeasonId` и `updatedAt`.

### Что считается прогрессом

Для MVP `CareerSave` хранит только верхнеуровневый курсор карьеры:

- `saveName`
- `selectedTeamId`
- `currentSeasonId`
- `currentDate`
- `currentRound`

А фактический прогресс сезона берётся из связанных сущностей:

- результаты матчей — из `Match`
- standings — из `Standing`

Подробная концепция сохранения описана в:

- [docs/career-save-concept-v1.md](/home/newuser/Documents/BM/docs/career-save-concept-v1.md:1)

## Relations

### Team -> Player

Связь:

- одна `Team` имеет много `Player`
- один `Player` может быть привязан к одной `Team`

В Prisma это выражено так:

- у `Team`: `players   Player[]`
- у `Player`: `teamId` и `team`

### Поведение связи

- `teamId` у игрока допускает `null`
- при удалении команды у игрока срабатывает `ON DELETE SET NULL`
- при обновлении `Team.id` действует `ON UPDATE CASCADE`

### Зачем это нужно

Такой вариант удобен для MVP, потому что:

- игрок может быть связан с командой;
- состав команды легко запрашивается по `teamId`;
- удаление команды не обязано удалять игроков физически.

## Constraints

Основные ограничения заданы не только в Prisma, но и на уровне SQL-миграции.

Файл с ограничениями:

- [prisma/migrations/20260430193143_add_team_and_player_models/migration.sql](/home/newuser/Documents/BM/prisma/migrations/20260430193143_add_team_and_player_models/migration.sql:1)

## Ограничения Team

- `shortName` уникален
- `name` не может быть пустой строкой
- `city` не может быть пустой строкой
- `shortName` не может быть пустой строкой
- `rating` должен быть в диапазоне `0..100`

### Практический смысл

Это защищает данные от:

- дублирования коротких названий команд;
- пустых текстовых полей;
- некорректного рейтинга команды.

---

## Ограничения Player

- `name` не может быть пустой строкой
- `age` должен быть в диапазоне `16..50`
- `shooting` в диапазоне `0..100`
- `passing` в диапазоне `0..100`
- `defense` в диапазоне `0..100`
- `rebounding` в диапазоне `0..100`
- `athleticism` в диапазоне `0..100`
- `potential` в диапазоне `0..100`
- `overall` в диапазоне `0..100`
- `overall <= potential`

### Практический смысл

Это защищает данные игрока от:

- нереалистичного возраста;
- атрибутов вне допустимого диапазона;
- ситуации, когда текущий `overall` выше, чем `potential`.

## Что важно понимать про текущую схему

На данный момент Prisma-схема покрывает только базовый слой данных MVP:

- команды;
- игроков;
- их связь.

Пока в схеме ещё нет:

- `Season`
- `Match`
- `Round`
- `Standings`
- `Career Save`

То есть текущая БД уже подходит для:

- командного и игрокового API;
- seed-данных;
- отображения состава;
- дальнейшего подключения simulation logic.

Но для полного MVP игрового цикла понадобятся следующие доменные сущности в следующих задачах.

## Итог

Текущая Prisma-схема MVP — это компактная и понятная база из `Team` и `Player`,
с жёсткими ограничениями на качество данных и простой связью `одна команда -> много игроков`.

Эта структура уже достаточна для:

- backend API команд и игроков;
- seed-данных;
- карточки команды и состава;
- подготовки к simulation engine и следующим сущностям сезона.
