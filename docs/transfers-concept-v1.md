# Transfer System Concept v1

## Цель документа

Этот документ фиксирует черновую концепцию трансферов для этапа `Post-MVP`.

Его задача:

- описать минимальные продуктовые правила покупки и продажи игроков;
- очертить будущие модели данных;
- зафиксировать ожидаемые изменения в roster flow;
- набросать UI-сценарий без обязательства реализовывать его в MVP;
- не привязывать команду к преждевременной реализации сложной экономики клуба.

---

## Статус

Текущий статус документа: `draft`.

Приоритет: `P2`  
Эпик: `Post-MVP Backlog`  
Компонент: `Future`  
Релиз: `Post-MVP`

Важно:

- трансферы не входят в MVP;
- отсутствие этой системы не блокирует релиз текущего сезона и career loop;
- документ нужен как опорная точка, чтобы после стабилизации MVP не начинать механику с нуля.

---

## Продуктовая роль трансферов

После MVP трансферы должны стать первой крупной менеджерской системой поверх уже работающего цикла:

```text
сезон -> межсезонье/окно -> изменение состава -> новый сезон
```

Смысл системы:

- дать пользователю рычаг влияния на состав;
- создать долгосрочные решения между сезонами;
- добавить движение игроков по лиге;
- усилить различия между сильными и слабыми командами не только через симуляцию матчей.

При этом первая версия трансферов должна оставаться достаточно простой:

- без сложных многосторонних обменов;
- без агентских переговоров;
- без длинной контрактной математики;
- без полной финансовой модели уровня sports sim AAA.

---

## Базовый принцип первой версии

Первая версия трансферов должна строиться вокруг управляемых сделок между командами лиги.

Базовая формула:

1. у игрока есть рыночная ценность;
2. у команды есть потребность в усилении или разгрузке состава;
3. в разрешённое окно команда может купить или продать игрока;
4. после сделки игрок меняет `teamId`;
5. новый roster участвует в следующем этапе сезона или в следующем сезоне.

Для `v1` рекомендуется опираться на сценарий:

- `покупка/продажа за value`,

а не на сценарий:

- `обмен игрок на игрок`,
- `обмен игрок + пик`,
- `трёхсторонний трейд`.

Это позволяет быстрее собрать работающую механику и не перегружать API.

## Transfer Window

### Зачем нужно окно

Окно ограничивает хаотичные сделки и делает механику предсказуемой.

Без окна трансферы будут:

- ломать понятный season loop;
- усложнять симуляцию и UI в любой момент;
- создавать слишком много внеплановых пересчётов состава и силы команд.

### Рекомендуемая первая версия

Для первой реализации достаточно двух состояний:

- `OPEN`
- `CLOSED`

И двух основных фаз:

1. предсезонное окно перед стартом нового сезона;
2. короткое внутрисезонное окно до условного дедлайна.

### Практическое правило v1

Для `Post-MVP` можно начать ещё проще:

- открыть трансферы только между сезонами.

Это самый безопасный старт, потому что:

- не нужно объяснять дедлайн внутри активного календаря;
- не нужно менять составы в середине уже симулируемого раунда;
- команда проще контролирует consistency season/save state.

### Что не нужно в первой версии

- ежедневный календарь трансферного окна;
- разные окна по лигам;
- отдельные правила free agency;
- срочные исключения и emergency-signing flow.

## Player Value

### Зачем нужен value

Система не должна принимать решение только по `overall`.

Если ориентироваться только на `overall`, то трансферы будут:

- слишком плоскими;
- плохо учитывать возраст и потенциал;
- плохо объяснять, почему один игрок дороже другого при близком текущем уровне.

### Базовые факторы value

Черновой набор факторов:

- `overall`
- `potential`
- `age`
- `position scarcity`
- `recent performance`
- `team need`

### Черновая логика

`PlayerValue` должен быть отдельной derived-метрикой, а не полем, которое правится вручную в каждом месте.

Упрощённая идея:

```text
value =
  current quality
  + future upside
  + positional demand
  - age decline pressure
  +/- recent form modifier
```

### Практический смысл факторов

`overall`

- даёт базовую рыночную силу игрока.

`potential`

- делает молодых игроков с ростом дороже, чем “готовых, но упёршихся в потолок”.

`age`

- снижает цену возрастных игроков и поддерживает долгосрочные решения по rebuild.

`position scarcity`

- позволяет не уравнивать полезного big man и очередного запасного guard при одинаковом `overall`.

`recent performance`

- даёт умеренную живость рынку после удачного или неудачного сезона.

`team need`

- не должен менять абсолютный `market value` игрока, но может менять готовность команды платить.

### Чего стоит избегать в v1

- очень шумной цены, которая прыгает после каждого матча;
- полного отделения value от атрибутов игрока;
- прямой зависимости только от одной цифры `overall`.

## Team Roster Changes

### Базовое правило

После подтверждённого трансфера игрок должен менять принадлежность команде через обычную связь `Player.teamId`.

Это значит:

- основная модель roster уже совместима с будущими трансферами;
- сама смена команды концептуально проста;
- сложность лежит не в поле `teamId`, а в правилах сделки, валидации и истории.

### Что должно происходить после сделки

После сделки система должна:

1. снять игрока со старой команды;
2. привязать игрока к новой команде;
3. сохранить запись о сделке;
4. обновить roster views;
5. при необходимости пересчитать derived team strength и related projections.

### Ограничения первой версии

- команда не должна уходить ниже минимального числа игроков в roster;
- команда не должна превышать максимальный размер roster;
- сделки внутри одного и того же окна должны быть атомарными;
- во время активной сделки игрок не должен продаваться дважды.

### Рекомендуемые рамки roster для v1

Можно держать простые ограничения:

- `minimum roster size`
- `maximum roster size`

без более сложных правил:

- homegrown quota;
- легионерский лимит;
- salary cap exceptions;
- two-way contracts.

## Product Rules Draft

Ниже предложен безопасный набор правил для первой рабочей версии.

### Сделка

Сделка состоит из:

- продающей команды;
- покупающей команды;
- одного игрока;
- agreed value;
- даты;
- статуса.

### Статусы сделки

Достаточно поддержать:

- `PROPOSED`
- `ACCEPTED`
- `REJECTED`
- `CANCELLED`
- `COMPLETED`

Для самой первой реализации можно даже начать без публичного `PROPOSED` flow и делать:

- мгновенное серверное решение `accepted/rejected`,

если это упростит механику AI-ответа команд.

### Правила принятия сделки AI-командой

Упрощённая логика принятия:

- сравнить `offered value` и `player market value`;
- учесть возраст и потенциал игрока;
- проверить, не ломает ли уход игрока минимальный roster;
- проверить, закрывает ли входящий ресурс потребности команды.

### Рекомендуемое поведение рынка

- слабые команды чаще продают возрастных игроков;
- сильные команды чаще покупают ready-now игроков;
- молодые и перспективные игроки реже доступны;
- franchise-level игроки должны иметь высокий порог недоступности.

## Data Model Draft

Ниже список моделей и полей, которые, вероятно, понадобятся после MVP.

Это не финальная схема Prisma, а черновик доменной структуры.

### `TransferWindow`

Назначение:

- хранить разрешённый период сделок.

Черновые поля:

- `id`
- `seasonId`
- `phase`
- `status`
- `opensAt`
- `closesAt`
- `createdAt`
- `updatedAt`

### `Transfer`

Назначение:

- хранить сам факт сделки и её жизненный цикл.

Черновые поля:

- `id`
- `seasonId`
- `windowId`
- `playerId`
- `fromTeamId`
- `toTeamId`
- `status`
- `agreedValue`
- `proposedAt`
- `resolvedAt`
- `completedAt`
- `createdAt`
- `updatedAt`

### `PlayerMarketSnapshot` или `PlayerValuation`

Назначение:

- фиксировать расчётную цену игрока на момент окна или сделки.

Черновые поля:

- `id`
- `playerId`
- `seasonId`
- `windowId`
- `marketValue`
- `overallFactor`
- `potentialFactor`
- `ageFactor`
- `demandFactor`
- `formFactor`
- `createdAt`

### Возможный `TeamBudget`

Назначение:

- ограничивать рынок, если команда решит добавить простую финансовую рамку.

Черновые поля:

- `id`
- `teamId`
- `seasonId`
- `transferBudget`
- `spentAmount`
- `receivedAmount`
- `createdAt`
- `updatedAt`

Важно:

- `TeamBudget` полезен, но не обязателен для самой первой версии трансферов;
- если команда захочет начать вообще без финансов, можно временно обойтись только правилами `team willingness` и `market value`.

### Возможный `TransferHistory` как отдельная сущность

Скорее всего не нужна.

Если `Transfer` уже хранит финальный статус и timestamps, отдельная история на старте даст лишнюю сложность.

Рекомендуемый подход:

- сначала считать `Transfer` одновременно и рабочей сделкой, и историей сделки;
- отдельный event log добавлять только если реально понадобится timeline переговоров.

## API Draft

Этот раздел не задаёт обязательный контракт, а только фиксирует возможное направление.

Примеры будущих endpoint-ов:

- `GET /transfers/windows/current`
- `GET /transfers/market`
- `GET /transfers/history`
- `POST /transfers`
- `POST /transfers/:id/cancel`

Примеры будущих сценариев:

- получить список игроков на рынке;
- отправить предложение по игроку;
- увидеть принято ли предложение;
- увидеть завершённые сделки сезона.

Для `v1` не стоит пытаться сразу сделать:

- длинные переговоры;
- встречные предложения;
- пакетные сделки;
- обмен несколькими игроками в одном запросе.

## UI Draft

### Основной пользовательский сценарий

Пользователь открывает раздел трансферов и видит:

- статус окна;
- доступных игроков;
- примерную стоимость;
- фильтры по позиции, возрасту, value и команде;
- свои активные и завершённые сделки.

### Рекомендуемые экраны

1. `Transfer Hub`

- статус окна;
- краткий бюджет или лимит;
- активные предложения;
- последние сделки по лиге.

2. `Transfer Market`

- таблица игроков;
- фильтры;
- карточка игрока;
- кнопка предложения сделки.

3. `Transfer History`

- завершённые переходы;
- бывшая и новая команда;
- цена;
- дата сделки.

### UI для первой версии

UI должен быть функциональным, а не “симулятором переговоров”.

Хороший первый вариант:

- таблица рынка;
- боковая панель игрока;
- простая форма `buy / cancel`;
- журнал завершённых сделок.

### Что не нужно в первой версии

- сложный inbox переговоров;
- сценки с агентами;
- многоэкранный wizard сделки;
- отдельная cinematic presentation трансфера.

## Save And Season Impact

Трансферы затрагивают не только `Team` и `Player`, но и `CareerSave`.

### Что важно сохранить

- активная карьера должна видеть уже обновлённый roster после сделки;
- следующий сезон должен стартовать с новым составом;
- история переходов не должна теряться после перезапуска.

### Что не нужно делать в лоб

Не стоит копировать весь roster внутрь `CareerSave`.

Лучше сохранить текущий принцип проекта:

- `CareerSave` хранит курсор карьеры;
- состав остаётся в доменных сущностях;
- трансферы меняют доменное состояние, которое save затем читает.

### Потенциальный риск

Если трансферы будут разрешены внутри активного сезона, появится вопрос:

- считаются ли уже сыгранные матчи частью старого состава, а будущие частью нового.

Поэтому для безопасного старта рекомендуется:

- запускать трансферы только между сезонами или между завершением сезона и созданием следующего.

---

## Out Of Scope For v1

Первая версия трансферов сознательно не должна включать:

- контракты с длиной и зарплатой;
- полноценный salary cap;
- обмены игрок на игрок;
- драфт-пики;
- аренды;
- buyout flow;
- свободных агентов как отдельный рынок;
- мораль игрока и список предпочтительных клубов;
- ручные переговоры по нескольким раундам.

Это всё может стать `v2+`, но не должно тормозить первую рабочую механику.

---

## Recommended Rollout

Оптимальный порядок внедрения после стабилизации MVP:

1. derived `player value`;
2. server-side transfer validation;
3. simple completed transfer flow between seasons;
4. transfer history UI;
5. optional market screen with proposals;
6. only after that consider budgets and intra-season window.

Такой rollout уменьшает риск:

- сначала появляется математика ценности;
- потом появляется запись сделки;
- только затем появляется более широкий продуктовый слой.

---

## Готовность

Эту задачу можно считать закрытой на уровне концепта, если команда согласна со следующими тезисами:

- трансферы остаются `Post-MVP`;
- первая версия строится вокруг простого окна и одиночной сделки по игроку;
- `Player.teamId` остаётся базовой точкой изменения состава;
- для полной реализации понадобятся как минимум `TransferWindow`, `Transfer`, `PlayerValuation`;
- UI первой версии должен быть табличным и прикладным, без тяжёлой переговорной драматургии.

---

## Итог

Трансферная система для `Post-MVP` должна быть не “большой футбольной экономикой”,
а аккуратным расширением уже работающей карьерной модели.

Если коротко:

```text
Сначала рынок и простые сделки.
Потом история и ограничения.
Потом более глубокий менеджмент.
```
