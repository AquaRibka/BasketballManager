# Career Save Concept v1

## Цель документа

Этот документ фиксирует минимальную концепцию сохранения карьеры для MVP-3.

Его задача:

- определить, что именно считается прогрессом карьеры;
- отделить собственно `save` от сезонных данных;
- зафиксировать минимальный набор полей для продолжения игры после перезапуска;
- не тащить в MVP лишние сложные слоты и дублирование состояния.

---

## Базовый принцип

`CareerSave` хранит не всю карьеру целиком в одном JSON-подобном объекте,
а только точку входа в текущее состояние карьеры.

То есть сохранение должно ответить на четыре вопроса:

1. как называется это сохранение;
2. какой командой управляет пользователь;
3. в каком сезоне находится карьера;
4. на какой дате/раунде нужно продолжать игру.

Все остальные данные прогресса берутся из связанных доменных сущностей:

- `Season`
- `Match`
- `Standing`

---

## Что считается прогрессом

Для MVP прогресс карьеры состоит из:

- выбранной команды;
- текущего сезона;
- текущего раунда;
- текущей игровой даты;
- уже сыгранных результатов матчей;
- текущих standings этого сезона.

Важно:

- `selected team`, `current season`, `current round`, `current date` хранятся прямо в `CareerSave`;
- `results` и `standings` не дублируются в `CareerSave`, а читаются из БД по `currentSeasonId`.

Это и есть ключевое правило концепции:

```text
CareerSave хранит курсор карьеры,
а не полную копию состояния сезона.
```

---

## Save Fields

Минимальный MVP-набор полей:

- `id`
- `saveName`
- `selectedTeamId`
- `currentSeasonId`
- `currentDate`
- `currentRound`
- `createdAt`
- `updatedAt`

### Значение полей

#### `saveName`

Человекочитаемое имя сохранения.

Нужно для:

- списка сохранений;
- выбора нужной карьеры в UI;
- различения нескольких прохождений.

#### `selectedTeamId`

Команда, которой управляет пользователь в этой карьере.

Нужно для:

- загрузки карьерного контекста;
- отображения активного клуба;
- фильтрации пользовательских сценариев в UI.

#### `currentSeasonId`

Сезон, к которому привязано текущее состояние карьеры.

Нужно для:

- загрузки календаря;
- загрузки матчей;
- загрузки standings;
- понимания, какой именно прогресс сейчас активен.

#### `currentDate`

Текущая игровая дата внутри карьеры.

Нужно для:

- восстановления точки продолжения;
- отображения календарного положения в UI;
- будущего перехода к date-based логике сезона.

#### `currentRound`

Текущий шаг сезона, на котором находится карьера.

Нужно для:

- восстановления текущего прогресса сезона;
- показа “следующего действия” после загрузки;
- связи с round-based MVP flow.

#### `createdAt`, `updatedAt`

Технические поля аудита.

Нужны для:

- сортировки сохранений;
- отображения “последнее обновление”;
- выбора самого свежего save.

## Career State

### Что хранится прямо в `CareerSave`

Это state верхнего уровня:

- идентичность сохранения;
- выбранная команда;
- активный сезон;
- текущий round/date курсор.

### Что НЕ хранится прямо в `CareerSave`

Это state доменных сущностей:

- расписание сезона;
- результаты отдельных матчей;
- сыгран ли конкретный матч;
- турнирная таблица;
- чемпион сезона;
- player box score;
- история прошлых сезонов.

Причина простая:

- эти данные уже имеют свои таблицы;
- дублирование в `CareerSave` создаст рассинхрон;
- в MVP важнее восстановить курсор прогресса, чем хранить snapshot всей лиги.

---

## Selected Team

`selectedTeamId` — обязательная часть сохранения.

### Почему это часть save, а не derived-поле

Потому что команда пользователя — это основа всей карьеры:

- она нужна сразу после загрузки;
- она определяет пользовательский контекст;
- она не должна вычисляться косвенно через сезон или standings.

### Что это НЕ означает

`selectedTeamId` не означает, что в save нужно копировать всю команду,
всех игроков или snapshot roster.

Для MVP достаточно:

- хранить ссылку на `Team`;
- подгружать актуальные связанные данные отдельно.

---

## Current Season

`currentSeasonId` — обязательная часть сохранения.

### Почему текущий сезон хранится явно

Потому что сохранение должно однозначно знать,
из какого сезона поднимать:

- календарь;
- матчи;
- standings;
- текущий round progression.

### Что это даёт

После перезапуска backend может:

1. загрузить `CareerSave`;
2. взять `currentSeasonId`;
3. получить все `Match` этого сезона;
4. получить `Standing` этого сезона;
5. продолжить игру с `currentRound/currentDate`.

---

## Match Results And Standings

### Результаты матчей

Результаты матчей считаются частью прогресса карьеры,
но физически живут в таблице `Match`.

Сохранение не должно хранить, например:

- массив `playedMatches`;
- массив `scores`;
- отдельный save-level snapshot расписания.

Вместо этого используется правило:

```text
Все результаты карьеры на текущем шаге
получаются запросом матчей по currentSeasonId.
```

### Standings

`standings` тоже считаются частью прогресса,
но живут в таблице `Standing`.

Сохранение не должно дублировать:

- wins;
- losses;
- pointDiff;
- positions;
- champion.

Вместо этого:

```text
CareerSave указывает на сезон,
а standings вычисляются/читаются как состояние этого сезона.
```

---

## Relations

Для MVP достаточно двух основных связей:

- `CareerSave -> Team`
- `CareerSave -> Season`

Это означает:

- `selectedTeamId` ссылается на `Team.id`
- `currentSeasonId` ссылается на `Season.id`

Этого достаточно, чтобы:

- создать сохранение;
- загрузить сохранение;
- обновлять прогресс;
- восстановить сезонный контекст после перезапуска.

---

## Что не обязательно в MVP

В MVP не нужны:

- несколько save slots в одной сложной структуре;
- ручной snapshot всей лиги внутри save;
- копия standings внутри save;
- копия календаря внутри save;
- отдельная история всех прошлых сезонов;
- rollback/branching сохранений;
- полное event-sourcing представление карьеры.

Это всё можно добавлять позже,
если появится реальная продуктовая необходимость.

---

## Практическое правило для реализации

Если коротко, то save должен быть минимальным:

```text
CareerSave =
  who the user controls +
  which season is active +
  where to resume progression
```

А сезонный прогресс восстанавливается через связанные таблицы:

```text
Season + Match + Standing
```

---

## Итог

Для MVP концепция сохранения такая:

- `CareerSave` хранит `saveName`, `selectedTeamId`, `currentSeasonId`, `currentDate`, `currentRound`, `createdAt`, `updatedAt`;
- выбранная команда входит в save напрямую;
- текущий сезон входит в save напрямую;
- результаты матчей и standings считаются частью прогресса, но хранятся в `Match` и `Standing`;
- этого достаточно, чтобы продолжить карьеру после перезапуска без лишнего дублирования данных.
