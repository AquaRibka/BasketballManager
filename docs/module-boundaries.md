# Module Boundaries

## Цель документа

Этот документ фиксирует границы ответственности между основными частями проекта:

- `backend`
- `frontend`
- `simulation engine`
- `shared`

Его задача — не допустить смешивания UI, API, бизнес-логики симуляции и общих типов,
а также зафиксировать логичное направление зависимостей в MVP.

---

## Общий принцип

Проект разделён на четыре слоя:

1. `frontend` — показывает данные и принимает действия пользователя;
2. `backend` — управляет состоянием игры, API и сохранением данных;
3. `simulation engine` — считает результаты матчей и игровые события;
4. `shared` — хранит общие типы, контракты и утилиты без привязки к UI и БД.

### Главная идея

- `frontend` не считает матч сам;
- `simulation engine` не знает о UI;
- `backend` не содержит сложную симуляцию внутри себя;
- `shared` не зависит от конкретного приложения.

---

## Схема зависимостей

Допустимое направление зависимостей:

```text
frontend -> shared
backend -> shared
backend -> simulation-engine
simulation-engine -> shared
```

Недопустимые зависимости:

```text
frontend -> backend (как импорт модуля)
frontend -> simulation-engine
shared -> backend
shared -> frontend
shared -> simulation-engine
simulation-engine -> backend
```

### Пояснение

- `frontend` общается с `backend` только через HTTP/API-контракт, а не через прямой импорт кода;
- `backend` может использовать `simulation-engine` как вычислительный модуль;
- `simulation-engine` может использовать только общие типы и утилиты из `shared`;
- `shared` должен оставаться самым нейтральным слоем.

---

## Backend Responsibilities

Пакет: `apps/backend`

### Backend отвечает за

- HTTP API;
- загрузку и сохранение данных;
- работу с базой данных;
- Prisma и репозитории;
- жизненный цикл карьеры и сезона;
- календарь, таблицу, команды, игроков;
- запуск симуляции через `simulation engine`;
- применение результатов симуляции к состоянию сезона;
- валидацию входных данных API;
- orchestration между базой, игровым состоянием и симуляцией.

### Backend не должен

- рендерить UI;
- содержать логику представления страниц;
- хранить сложную матчевую симуляцию прямо внутри контроллеров и сервисов API;
- зависеть от frontend-кода;
- дублировать типы, которые уже есть в `shared`.

### Примеры backend use cases

- создать карьеру;
- выбрать активную команду пользователя;
- вернуть состав команды;
- вернуть календарь текущего раунда;
- запустить симуляцию матча или раунда;
- обновить турнирную таблицу;
- завершить сезон.

---

## Frontend Responsibilities

Пакет: `apps/frontend`

### Frontend отвечает за

- экраны и навигацию;
- пользовательский поток карьеры;
- отображение команды, игроков, календаря и таблицы;
- формы и действия пользователя;
- запросы к backend API;
- локальное UI-состояние;
- клиентскую валидацию для удобства пользователя.

### Frontend не должен

- напрямую ходить в базу данных;
- содержать Prisma;
- считать результаты симуляции матча;
- хранить “истинное” состояние сезона отдельно от backend;
- импортировать код backend;
- принимать продуктовые решения за сервер, например самовольно продвигать раунд.

### Что frontend делает в MVP

- показывает экран старта карьеры;
- показывает экран команды;
- показывает календарь;
- даёт кнопку симуляции;
- показывает результаты раунда;
- показывает турнирную таблицу;
- даёт перейти к следующему раунду.

---

## Simulation Responsibilities

Пакет: `packages/simulation-engine`

### Simulation engine отвечает за

- расчёт результата матча;
- применение игровых правил симуляции;
- вычисление счёта;
- вычисление победителя;
- при необходимости генерацию игровых событий и статистики;
- deterministic или pseudo-random матчевую логику.

### Simulation engine не должен

- знать про HTTP;
- знать про React;
- знать про Prisma и SQL;
- сам читать или писать в базу данных;
- хранить карьеру, сезон или таблицу как постоянное состояние;
- зависеть от backend-контекста исполнения.

### Что simulation engine получает на вход

- описание команд;
- описание игроков;
- исходные параметры матча;
- seed/настройки симуляции, если это нужно.

### Что simulation engine возвращает

- счёт матча;
- победителя;
- проигравшего;
- при необходимости статистику и игровые события.

### Практическая роль в MVP

`simulation-engine` — это чистый вычислительный модуль.
`backend` вызывает его, получает результат и уже сам решает, как сохранить этот результат в БД.

---

## Shared Types

Пакет: `packages/shared`

### Shared отвечает за

- общие типы домена;
- DTO-контракты между frontend и backend;
- enum-значения;
- константы, которые не должны дублироваться;
- простые чистые утилиты без инфраструктурной зависимости;
- общие схемы, если команда позже решит использовать единый validation layer.

### Что может лежать в shared

- `PlayerPosition`;
- типы `Team`, `Player`, `Match`, `StandingsRow`;
- DTO для `CreateCareer`, `SimulateRound`, `TeamDetails`;
- константы лимитов и диапазонов;
- базовые helper-функции форматирования или маппинга без побочных эффектов.

### Shared не должен

- содержать React-компоненты;
- содержать Prisma models или запросы к БД;
- содержать HTTP-контроллеры;
- содержать тяжёлую матчевую симуляцию;
- знать о storage, runtime и окружении конкретного приложения.

---

## Граница между Backend и Simulation Engine

Это одна из самых важных границ проекта.

### Backend делает

- определяет, какой матч нужно симулировать;
- загружает команды и игроков из БД;
- формирует входные данные для симуляции;
- вызывает `simulation engine`;
- сохраняет результат;
- обновляет таблицу и состояние сезона.

### Simulation engine делает

- принимает входные данные матча;
- вычисляет результат;
- возвращает результат в чистом виде.

### Принцип

Если код отвечает на вопрос:

- `что нужно симулировать и куда сохранить результат?` — это `backend`
- `как именно рассчитывается исход матча?` — это `simulation engine`

---

## Граница между Frontend и Backend

### Backend

- источник истины для состояния игры;
- владелец бизнес-правил сезона;
- владелец данных и доступа к БД.

### Frontend

- потребляет API;
- показывает состояние пользователю;
- отправляет команды пользователя на сервер.

### Принцип

Если код отвечает на вопрос:

- `что пользователь видит и куда нажимает?` — это `frontend`
- `что означает это действие для карьеры и сезона?` — это `backend`

---

## Что важно не смешивать в MVP

### Нельзя

- писать логику симуляции прямо в React-компонентах;
- писать SQL/Prisma в `simulation-engine`;
- выносить доменные типы одновременно в `backend` и `frontend` копиями;
- хранить расчёт таблицы частично в UI, частично в backend без явной границы;
- импортировать backend-модули во frontend;
- смешивать DTO API и внутренние структуры симуляции без явного преобразования.

### Нужно

- держать domain contracts централизованно;
- вызывать симуляцию только из backend;
- использовать `shared` для типов и общих констант;
- хранить долгоживущее состояние сезона в backend и базе.

---

## Практическое распределение по текущей структуре

### `apps/backend`

- API endpoints;
- Prisma integration;
- career services;
- standings services;
- season progression;
- orchestration симуляции.

### `apps/frontend`

- страницы;
- компоненты;
- клиентские сценарии;
- запросы к API;
- UI-состояние.

### `packages/simulation-engine`

- pure simulation logic;
- расчёт матчей;
- генерация результатов.

### `packages/shared`

- общие типы;
- enums;
- DTO;
- constants;
- simple pure utilities.

---

## Итоговое правило для команды

```text
Frontend показывает и отправляет действия.
Backend управляет сезоном, API и данными.
Simulation engine считает матч.
Shared хранит общие типы и контракты.
```
