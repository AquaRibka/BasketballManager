# Prisma Migrations Guide

## Цель документа

Этот документ фиксирует, как в проекте Basketball Manager:

- создавать миграции Prisma;
- применять их локально;
- сбрасывать локальную БД в чистое состояние;
- заново заполнять её seed-данными;
- называть миграции единообразно.

Главная цель — убрать ручной хаос при изменении схемы БД.

## Основной workflow

Обычный порядок работы разработчика такой:

1. изменить `prisma/schema.prisma`
2. создать и применить новую миграцию
3. при необходимости перегенерировать Prisma Client
4. заполнить БД seed-данными
5. проверить backend или тесты

---

## Создание и применение миграции

### Основная команда

```bash
npx prisma migrate dev --name <migration_name>
```

Пример:

```bash
npx prisma migrate dev --name add_player_stamina
```

### Что делает команда

- сравнивает текущую схему с БД;
- создаёт новую папку миграции в `prisma/migrations`;
- применяет миграцию к локальной БД;
- обновляет `_prisma_migrations`;
- при необходимости обновляет Prisma Client.

### Когда использовать

Использовать каждый раз, когда вы меняете:

- модели Prisma;
- поля;
- enum;
- relation;
- constraint;
- индекс;
- default value.

## Команды проекта

### Применить новую локальную миграцию

```bash
npm run prisma:migrate -- --name <migration_name>
```

Пример:

```bash
npm run prisma:migrate -- --name add_team_logo
```

### Перегенерировать Prisma Client

```bash
npm run prisma:generate
```

### Заполнить БД seed-данными

```bash
npm run prisma:seed
```

или

```bash
npm run db:seed
```

### Полный reset локальной БД

```bash
npm run prisma:reset
```

Что делает reset:

- удаляет все данные;
- заново применяет все миграции;
- запускает seed-команду.

---

## Когда использовать reset

`prisma migrate reset` — это локальная developer-команда.

Её использовать нормально, если:

- схема сломалась в процессе разработки;
- нужно быстро получить чистую БД;
- seed-данные устарели;
- нужно повторить onboarding flow с нуля.

### Важно

`reset` полностью уничтожает локальные данные в БД.

Поэтому:

- использовать только для локальной разработки;
- не использовать как способ “отката” на shared/production базе.

---

## Как откатывать изменения

В Prisma для локальной разработки практический “откат” обычно делается не через отдельный rollback-файл,
а через один из двух путей:

### Вариант 1. Локально пересобрать состояние

Если изменения ещё не ушли далеко:

1. поправить `schema.prisma`
2. выполнить `npm run prisma:reset`
3. при необходимости заново выполнить `npm run db:seed`

### Вариант 2. Сделать новую корректирующую миграцию

Если миграция уже стала частью истории проекта:

1. не удалять старую миграцию руками;
2. изменить `schema.prisma` в нужную сторону;
3. создать новую миграцию через `migrate dev`;
4. закоммитить именно новую корректирующую миграцию.

### Главное правило

Не редактировать уже применённые миграции задним числом без очень веской причины.

Для командной работы безопаснее:

- оставить историю миграций линейной;
- исправлять схему следующей миграцией.

## Naming rules

Названия миграций должны быть:

- короткими;
- понятными;
- в `snake_case`;
- ориентированными на смысл изменения.

### Хорошие примеры

- `add_player_stamina`
- `create_season_table`
- `add_match_status_enum`
- `link_players_to_team`
- `add_team_short_name_unique`

### Плохие примеры

- `init2`
- `fix`
- `changes`
- `update`
- `test_migration`

### Рекомендация

Использовать шаблон:

```text
<verb>_<entity>_<change>
```

Например:

- `add_player_contract_fields`
- `create_standings_table`
- `update_team_rating_constraints`

## Seed workflow

После `reset` или после поднятия новой локальной БД нужно убедиться, что данные снова заполнены.

Текущий seed-скрипт:

- [prisma/seed.mjs](/home/newuser/Documents/BM/prisma/seed.mjs:1)

Он:

- создаёт минимальную лигу для MVP;
- создаёт команды;
- создаёт составы игроков;
- работает как dev baseline для API и simulation.

### Проверка

После seed ожидается, что:

- команды существуют;
- у команд есть игроки;
- backend `teams` и `players` endpoint-ы возвращают данные.

---

## Что не делать

- не менять структуру БД руками через SQL, если то же самое можно выразить в Prisma schema;
- не создавать миграции с бессмысленными именами;
- не править старые миграции ради “красоты”, если они уже были применены;
- не использовать `reset` как production rollback;
- не забывать про seed после полного сброса базы.

---

## Минимальный рабочий сценарий

Если упростить всё до одного практического сценария:

1. изменить [prisma/schema.prisma](/home/newuser/Documents/BM/prisma/schema.prisma:1)
2. выполнить:

```bash
npm run prisma:migrate -- --name <migration_name>
```

3. при необходимости выполнить:

```bash
npm run prisma:seed
```

4. проверить backend

Если локальная БД развалилась:

```bash
npm run prisma:reset
```

---

## Итог

В проекте Prisma-модели меняются только через `schema.prisma`,
а состояние БД обновляется через нормальный migration workflow:

- `migrate dev` для новых изменений;
- `reset` для локального полного пересборa;
- `seed` для dev-данных;
- понятные `snake_case` имена миграций для чистой истории.
